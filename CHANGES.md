# CHANGES — from QZ Tray v2 (`qz-tray.js`) to v3.0-alpha (`src-next/`)

This document explains the differences between the **old v2 library** and the new
**3.0 alpha build** that lives under `src-next/`, and walks through the new
TypeScript structure **file by file**.

---

## 1. The three "eras" of the code

To get your bearings, three representations of the library coexist in this
repository:

| What | Where | What it is |
|------|------|------------|
| **Monolithic v2** | `qz-tray.js` (root, ~153 KB) | The historical library: a **single IIFE file** `var qz = (function(){ … })()`. Contains polyfills, the private `_qz` object, the public `qz` object — all in JavaScript with JSDoc. |
| **Modular v2** | `src/**/*.js` | Exactly the same behavior as v2, but **split into ES modules** (`src/internal/*` = former `_qz`, `src/api/*` = former `qz`). Still a **global singleton** with JSDoc. Bundled to `dist/qz-tray.js`. |
| **v3.0 alpha** | `src-next/**/*.ts` | The **TypeScript rewrite**, **instance-based**: no more mandatory global singleton. Bundled to `dist/qz-next.js` / `.mjs`. |

> In short: `qz-tray.js` and `src/` are **the same v2** (one flattened, one
> modular). `src-next/` is **v3**. This document mainly compares
> **v2 (qz-tray.js / src) ⇄ v3 (src-next)**.

---

## 2. Fundamental conceptual differences

### 2.1 From global singleton to instance factory

**Before (v2):** there was *only one* `qz`. All state (connection, certificate,
signing, print defaults, callbacks) lived in two module-level global objects:
`_qz` (internal/private) and `qz` (public). Configuring the library meant
**mutating those globals**:

```js
qz.security.setCertificatePromise(...);
qz.security.setSignaturePromise(...);
qz.configs.setDefaults({ copies: 2 });
qz.websocket.connect();
```

Consequence: it was **impossible** to have two clients on the same page (e.g. two
different certificates, or two different QZ hosts) — they trampled each other.

**Now (v3):** a **factory** `createQz(options)` returns a **fully independent**
client, with its own state isolated in a `Context` object. You can create as many
instances as you like:

```ts
import { createQz } from 'qz-tray';

const qz = createQz({
    security: { certificate: () => myCert, sign: (s) => mySign(s) },
    defaults: { copies: 2 },
    connect: { host: 'localhost' },
});
await qz.connect();
```

Configuration is **declared up front** at creation time, instead of being
accumulated through mutating calls on a global.

### 2.2 Immutable configuration

**Before:** `qz.configs.create(...)` returned a **mutable** `Config` object;
`reconfigure()` / `setPrinter()` modified `this.config` in place.

**Now:** `PrintConfig` is an **immutable** (readonly) value. The methods
`.with(patch)` and `.withPrinter(p)` **do not mutate**: they return a *new*
config. This makes configuration predictable and shareable without side effects.

### 2.3 JSDoc → real TypeScript types

**Before:** parameter documentation lived in huge `@param options.*` JSDoc
blocks, **not verified** by the compiler.

**Now:** the entire public "contract" is in `src-next/types.ts` as TypeScript
interfaces **checked by the compiler**. Malformed argument shapes are caught at
build time, not at runtime.

### 2.4 Dependency injection instead of global access

**Before:** every internal function read the `_qz` / `qz` globals directly.

**Now:** each module receives its dependencies explicitly
(`ctx`, `security`, `transport`). No implicit state → everything is
**testable in isolation** and free of hidden singletons.

### 2.5 Guaranteed backwards compatibility

v3 **does not break** existing code. The `legacy.ts` file reconstructs the
**identical API surface** of v2 (`qz.websocket.connect`, `qz.configs.create`,
`qz.print`, `qz.security.*`, …) on top of *one* default instance. So:

- `import qz from 'qz-tray'` → the old global singleton, API-compatible.
- `import { createQz } from 'qz-tray'` → the new instance-based model.

### 2.6 Toolchain

| | v2 | v3 |
|--|----|----|
| Language | JS + JSDoc | TypeScript (`strict`) |
| Types | none (JSDoc only) | `.d.ts` generated by `tsc` |
| Bundler | esbuild → `dist/qz-tray.*` | esbuild → `dist/qz-next.*` |
| Output | IIFE/UMD + ESM | IIFE/UMD + ESM (identical scheme) |
| Runtime version | 2.2.6 | 2.2.6 (protocol unchanged; "3.0" is the library) |

> Note: the `VERSION` in the `Context` is still `2.2.6` because it identifies
> **protocol compatibility with QZ Tray**, not the version of the JS library.

## 3. The new `src-next/` structure

```
src-next/
├── index.ts              ← public entry point (createQz + legacy default)
├── client.ts             ← the createQz() factory
├── legacy.ts             ← backwards-compatible v2 singleton
├── types.ts              ← all public interfaces
│
├── core/                 ← client engine (one per instance)
│   ├── context.ts        ← per-instance state container
│   ├── transport.ts      ← WebSocket + QZ protocol
│   ├── security.ts       ← certificate + signing
│   ├── state.ts          ← connection state + version comparison
│   └── compatible.ts     ← compatibility shims for older QZ versions
│
├── printing/             ← printing
│   ├── config.ts         ← immutable PrintConfig
│   ├── defaults.ts       ← default print options
│   └── print.ts          ← qz.print dispatch
│
├── namespaces/           ← the "namespace" APIs (qz.printers, qz.usb, …)
│   ├── api.ts            ← qz.api (diagnostics/compat)
│   ├── printers.ts       ← qz.printers
│   ├── networking.ts     ← qz.networking
│   ├── serial.ts         ← qz.serial
│   ├── socket.ts         ← qz.socket
│   ├── usb.ts            ← qz.usb
│   ├── hid.ts            ← qz.hid
│   └── file.ts           ← qz.file
│
└── internal/             ← pure, stateless utilities
    ├── util.ts           ← pure helpers (former helpers.js + parts of tools.js)
    ├── sha.ts            ← SHA-256
    ├── streams.ts        ← callback-stream types/dispatch
    └── log.ts            ← per-instance logger
```

**Mapping to v2:** `core/` + `internal/` replace `src/internal/*`
(former `_qz`); `namespaces/` + `printing/` replace `src/api/*` (former `qz`).

## 4. File-by-file walkthrough

### Entry point and composition

#### `index.ts`
The package entry point (`main`/`module` in `package.json` point to the bundle
generated from here). Exposes both usage models:
- **named export** `createQz` + the public types (`Qz`, `BoundConfig`,
  `PrintConfig`, and all of `types.ts`) → modern instance-based usage;
- **default export** `qz` (from `legacy.ts`) → the backwards-compatible singleton.

#### `client.ts` — the `createQz()` factory
The heart of the new architecture. `createQz(options)`:
1. creates an isolated `Context` (the instance state);
2. builds `security` and `transport` on top of that context;
3. applies any connection defaults;
4. composes and returns the public client object, with:
   - connection methods (`connect`, `disconnect`, `isActive`, …);
   - printing (`config()`, `setDefaults()`, `print`);
   - `security` (runtime overrides);
   - all the **namespaces** (`printers`, `api`, `networking`, `serial`,
     `socket`, `usb`, `hid`, `file`);
   - `_ctx`, an escape hatch to the context (advanced/testing use).

   It also defines `BoundConfig`: a `PrintConfig` enriched with a "bound"
   `.print()` and the chainable builders `.with()` / `.withPrinter()`, so you can
   write `qz.config('PDF-XChange').with({copies:2}).print(data)`.

#### `legacy.ts` — backwards-compatible singleton
Reconstructs the old global `qz.*` API **exactly**, on top of **one**
`createQz()` instance. It reintroduces here the **mutable** v2 `Config` object
(`setPrinter` / `reconfigure` / `getOptions` / `print`) as a *façade* over the
new immutable `PrintConfig`, keeping the current "snapshot" internally. It also
preserves the historical positional signatures of `qz.print(...)` and the
deprecated members (e.g. `websocket.getNetworkInfo` → `networking.device`).
Thanks to this file, existing v2 code keeps working unchanged.

#### `types.ts` — public type surface
Replaces the large `@param options.*` JSDoc blocks with compiler-checked TS
interfaces: `QzOptions`, `SecurityOptions`, `ConnectOptions`, `PrintOptions`,
`PrinterInput`/`Printer`, `PrintData`/`PrintPayload`, the types for
`Promiser`/`Hasher`/`WebSocketCtor`, the `LnaShim` (Local Network Access), etc.
It is the "contract" for whoever consumes `createQz(...)`.

---

### `core/` — the engine (one per instance)

#### `core/context.ts`
The **per-instance state container**: the single object that replaces **both** of
v2's global singletons (`_qz` internal **and** `qz` public). It holds:
`TITLE`/`VERSION`/`DEBUG`, the logger, the injectable providers
(`ws`/`promise`/`hash`/`lna`), the current connection, the pending calls
(uid→promise map), the `connectConfig` (host/ports/protocol/keepAlive/retry),
the callback registries (error/closed/stream), and the print `defaultConfig`.
`createContext()` initializes it, with auto-detection of `WebSocket` and of the
global `lna` helper. Because every `createQz()` creates its own `Context`,
**two clients coexist** on the same page — impossible with the old globals.

#### `core/transport.ts`
A faithful port of the `_qz.websocket` + `qz.websocket` protocol logic, but every
reference to the singleton is replaced by the injected `ctx` and `security`. It
handles: WebSocket opening (with the LNA shim and "denied" handling), the retry
loop over **host × ports × secure/insecure**, keep-alive (ping), routing of
incoming messages (per-uid responses vs stream events), the `getVersion`
handshake + certificate send, automatic **signing** of privileged calls
(hash → sign → send), and `dataPromise()` — the method all namespaces use to make
a call and get the response as a Promise. It exposes the connection API
(`connect`/`disconnect`/`getConnectionInfo`/`setUsingSurf`/…).

#### `core/security.ts`
**Certificate + signing** handling for a single instance: what in v2 was the
mutable global state `_qz.security`. It accepts both the **modern** provider shape
(`() => value | Promise`, via `SecurityOptions`) and the **legacy** shapes
(`setCertificatePromise` / `setSignaturePromise` with a resolver or
async-factory), normalizing them. It contains the `UNDIALOGED` list of calls that
do **not** require signing (e.g. `getVersion`, `printers.getStatus`, …) via
`needsSigned()`, and manages the signing algorithm (`SHA1`/`SHA256`/`SHA512`).

#### `core/state.ts`
Helpers that read the **live** `ctx` state, kept separate from the transport to
**avoid circular dependencies** (security/compatible use them without depending on
the transport): `isActive()` (connection alive?), `assertActive()` (throws a
clear error if not connected), and the version comparison `versionCompare()` /
`isVersion()` against the connected QZ version.

#### `core/compatible.ts`
The **backwards-compatibility transforms** applied based on the connected QZ Tray
version (former `_qz.compatible`), but with an explicit `ctx`:
- `compatData()`: converts print payloads for older QZ versions
  (Uint8Array→base64/hex, `dotDensity` normalization, v2.0 format conversion);
- `compatConfig()`: adapts config options to older versions
  (`rasterize`, `forceRaw`→`altPrinting`, `spool`→`perSpool`/`endOfDoc`) operating
  on a clone;
- `compatAlgorithm()`: warns if the connected version does not support alternate
  signing algorithms.

---

### `printing/` — printing

#### `printing/config.ts`
The **immutable** `PrintConfig` that replaces the v2 `Config` constructor (which
mutated `this.config` / `this._dirtyOpts`). It is a readonly value
(`printer` + `options` + `dirty`); `.with(patch)` and `.withPrinter(p)` return
**new** configs. It tracks `dirty` (the keys *explicitly* set by the user) to
drive the version shims — defaults inherited at creation time do **not** count as
"dirty".

#### `printing/defaults.ts`
The **default print options** for new configs (former
`_qz.printing.defaultConfig`): `colorType`, `copies`, `density`, `duplex`,
`margins`, `orientation`, `rasterize`, `size`, `units`, etc.

#### `printing/print.ts`
A faithful port of `qz.print`, but operating on immutable `PrintConfig` values and
the injected transport. It normalizes configs/data/signatures/timestamps into
parallel arrays, applies `makeRelativePathsAbsolute` + `compatData` to the
payloads, and builds a **chain** of promises (instead of `Promise.all`) so that,
with `resumeOnError`, the errors of individual jobs are **collected** and
re-thrown together at the end.

---

### `namespaces/` — the namespace APIs

Each file exports a `create<Name>(ctx, transport)` factory that returns the
namespace object. Almost all methods are "thin": they prepare the parameters and
delegate to `transport.dataPromise(call, params)`.

#### `namespaces/api.ts` — `qz.api`
**Diagnostics/compatibility** calls: `showDebug`, `getTitle`/`setTitle`,
`getVersion`, the `isVersion`/`isVersionGreater`/`isVersionLess` checks, and the
overrides for the injectable providers (`setPromiseType`, `setSha256Type`,
`setWebSocketType`).

#### `namespaces/printers.ts` — `qz.printers`
Printer management: `getDefault`, `find`, `details`, `startListening`/
`stopListening` (with `jobData`/`maxJobData`/`flavor` options), `clearQueue`,
`getStatus`, and `setPrinterCallbacks` to register status/job callbacks.

#### `namespaces/networking.ts` — `qz.networking`
Network info (`device`/`devices`/`hostname`) with **version shims**: on QZ 2.0 it
falls back to `websocket.getNetworkInfo`, remapping the fields, and on versions
< 2.2.2 it derives the hostname from `device`.

#### `namespaces/serial.ts` — `qz.serial`
Serial ports (RS232): `findPorts`, `openPort`, `sendData` (converting a `FILE`
path to absolute on the versions that support it), `closePort`, and
`setSerialCallbacks`.

#### `namespaces/socket.ts` — `qz.socket`
Raw TCP sockets: `open`, `close`, `sendData` (normalizing the payload), and
`setSocketCallbacks`.

#### `namespaces/usb.ts` — `qz.usb`
USB devices: enumeration (`listDevices`/`listInterfaces`/`listEndpoints`),
claim/release, I/O (`sendData`/`readData`), and streaming
(`openStream`/`closeStream`), plus `setUsbCallbacks`. It uses
`normalizeDeviceInfo` to accept **either** a device-info object **or** the old
positional arguments (vendorId, productId, …), preserving v2 compatibility.

#### `namespaces/hid.ts` — `qz.hid`
HID devices: symmetric to `usb.ts` (listDevices, claim/release, sendData/
readData, streams, feature reports). It also handles the case of older versions
that only accept `PLAIN` data, explicitly rejecting unsupported formats.

#### `namespaces/file.ts` — `qz.file`
QZ-side filesystem access: `list`, `read`, `write`, `remove`, `startListening`
(with `include`/`exclude` normalized to arrays)/`stopListening`, and
`setFileCallbacks`. The `withPath` helper merges the path into the parameters.

---

### `internal/` — pure, stateless utilities

#### `internal/util.ts`
**Pure, instance-agnostic** helpers (former `helpers.js` + the pure parts of
`tools.js`), therefore trivially testable: `ensureArray`, `normalizeData`,
`normalizePrinter`, `normalizeDeviceInfo` (the positional shim for USB/HID),
`stringify` (JSON that skips `promise` keys and neutralizes Prototype.js's
`Array#toJSON`), `deepExtend` (deep merge), the `uint8ToHex`/`uint8ToBase64`
conversions, the "surf" logic (`isQualified`/`appendSurf`), `newUID` (id to map
response↔call), and `absolute`/`makeRelativePathsAbsolute` (file/image path
resolution).

#### `internal/sha.ts`
A pure **SHA-256** implementation (verbatim port of `_qz.SHA`, adapted from Chris
Veness's code, MIT). No singleton: it is the context's default hash function,
overridable via `hasher`.

#### `internal/streams.ts`
The **stream** type identifiers (`SERIAL`/`USB`/`HID`/`PRINTER`/`FILE`/`SOCKET`)
and the `dispatch()` helper that invokes registered callbacks, tolerating an
array / a single function / unset.

#### `internal/log.ts`
A **per-instance logger**: `trace`/`info`/`warn`/`allay`/`error`, with `trace` and
`allay` (debug warnings) gated by the instance's `debug` flag — no more
module-global `DEBUG` flag.

## 5. How it builds

- `npm run build:next` → `dist/qz-next.js` (IIFE/UMD) + `dist/qz-next.mjs` (ESM),
  via `scripts/build-next.mjs` (esbuild compiles the TS with type-stripping).
- `npm run typecheck` → type checking (`tsc --noEmit`).
- `npm run types` → generates the `.d.ts` files in `dist/types/` (see
  `tsconfig.json`).
- `npm run build:all` → v2 build + v3 build + types.

The v3 bundle produces the **same output scheme** as v2 (browser global +
CommonJS + AMD + ESM), so it is a drop-in from a distribution standpoint.

---

## 6. Summary for migrators

- **Existing code:** no changes needed — `import qz from 'qz-tray'` keeps working
  (via `legacy.ts`).
- **New code:** prefer `createQz(options)`, configuring security/defaults/connect
  up front, and use the immutable `PrintConfig` with `.with()` / `.withPrinter()`.
- **Main benefit:** multiple independent instances, isolated state, checked types,
  modules testable without globals.
